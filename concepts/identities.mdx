A ZeroTier identity is the cryptographic foundation of everything in the network. Every node has exactly one identity, which simultaneously serves as its authentication credential, its signing key, and the source of its network address. Understanding identities is essential for operating ZeroTier securely.

## What Is a ZeroTier Identity?

An identity is a **Curve25519 / Ed25519 ECC key pair** coupled with a **40-bit address** derived from the public key via a memory-hard proof-of-work function.

From `node/Identity.hpp`:

```cpp
/**
 * An identity consists of a public key, a 40-bit ZeroTier address computed
 * from that key in a collision-resistant fashion, and a self-signature.
 *
 * The address derivation algorithm makes it computationally very expensive to
 * search for a different public key that duplicates an existing address.
 */
class Identity {
    // ...
    Address _address;        // 40-bit derived address
    ECC::Public _publicKey;  // Curve25519/Ed25519 public key set
    ECC::Private* _privateKey; // Private key (null if public-only)
};
```

The address is 40 bits wide, displayed as a 10-character hex string (e.g. `a1b2c3d4e5`). It is **not** a UUID, IP address, or random number — it is a deterministic fingerprint of the public key.

## Address Derivation: How It Works

Address generation in `Identity::generate()` (`node/Identity.cpp:84`) executes the following algorithm:

<Steps>
  <Step title="Generate a key pair">
    A fresh Curve25519/Ed25519 key pair is generated randomly using a secure RNG.
  </Step>
  <Step title="Run the memory-hard hash">
    The public key is fed into `_computeMemoryHardHash()`, a custom function that composes **SHA-512** and **Salsa20/12** over a **2 MiB working buffer** (`ZT_IDENTITY_GEN_MEMORY = 2097152` bytes). This makes the computation both CPU- and memory-intensive, which raises the cost of brute-force address mining.

    ```cpp
    // From node/Identity.cpp
    #define ZT_IDENTITY_GEN_HASHCASH_FIRST_BYTE_LESS_THAN 17
    #define ZT_IDENTITY_GEN_MEMORY 2097152
    ```

    The Salsa20 stream is initialized from the SHA-512 digest and expanded in a CBC-like sequential manner (not randomly seekable), then used as a look-up table to produce a final 64-byte digest.
  </Step>
  <Step title="Check the hashcash condition">
    The first byte of the digest must be less than `17` (out of 255). This succeeds roughly 1 in 15 times, so on average 15 key pairs must be generated per identity. On a slow embedded system this can take 10 seconds or more.
  </Step>
  <Step title="Extract the address">
    The **last 5 bytes** (bytes 59–63) of the passing digest become the 40-bit ZeroTier address. Reserved addresses are excluded and cause a retry.

    ```cpp
    _address.setTo(digest + 59, ZT_ADDRESS_LENGTH); // last 5 bytes
    ```
  </Step>
</Steps>

### Why Generation Can Be Slow

The memory-hard proof-of-work serves two purposes:

1. **Anti-spoofing**: It makes it computationally expensive to generate a key pair that hashes to a specific target address. An attacker cannot cheaply mint an address that collides with yours.
2. **Anti-Sybil**: Mass-generating throwaway identities is expensive, which limits spam and DoS using freshly created addresses.

<Tip>
  Normal identity generation takes under a second on modern hardware. On slow or embedded hardware it may take up to 30 seconds. This is a one-time cost — the generated identity is saved to disk and reused indefinitely.
</Tip>

### Validation

Any node can independently verify that a given identity is legitimate by calling `Identity::locallyValidate()` (`node/Identity.cpp:104`), which re-runs the memory-hard hash on the public key and checks that:

- The first byte of the digest is `< 17`
- Bytes 59–63 of the digest match the claimed address

This means there is no need to trust any server when verifying an identity — validation is entirely local.

## Identity Files

ZeroTier stores identities in two files in its working directory (typically `/var/lib/zerotier-one/` on Linux or `/Library/Application Support/ZeroTier/One/` on macOS).

<Tabs>
  <Tab title="identity.secret">
    Contains the **full identity**: address, public key, and private key. The string format is colon-separated:

    ```
    <address>:0:<public-key-hex>:<private-key-hex>
    ```

    For example:
    ```
    a1b2c3d4e5:0:1a2b3c...f0:9e8d7c...01
    ```

    The `0` in the second field is the identity type (currently always `0` for Curve25519/Ed25519).

    This file is generated by `Identity::generate()` and serialized via `Identity::toString(true, buf)` (`node/Identity.cpp:121`). The private key portion is only written when `includePrivate = true`.
  </Tab>
  <Tab title="identity.public">
    Contains the **public portion only**: address and public key, no private key. The format is identical but without the trailing private key field:

    ```
    <address>:0:<public-key-hex>
    ```

    This file is safe to share with others. The network controller uses it to verify credentials. Other peers receive it during HELLO exchange.
  </Tab>
</Tabs>

<Warning>
  `identity.secret` is the most sensitive file on a ZeroTier node. Whoever possesses it can impersonate your node on any network and decrypt all traffic encrypted to your address. Treat it like an SSH private key or a TLS certificate private key.

  For network controllers, the risk is compounded: the controller's `identity.secret` is the private key for the certificate authority that signs all member COMs on every network the controller manages.
</Warning>

## Managing Identities with zerotier-idtool

The `zerotier-idtool` command-line utility provides direct access to identity operations without needing a running ZeroTier service.

<CodeGroup>
  ```bash Generate a new identity
  # Generates identity.secret and identity.public in current directory
  zerotier-idtool generate identity.secret identity.public
  ```

  ```bash Validate an identity
  # Returns 0 (exit code) if valid, non-zero if invalid
  zerotier-idtool validate identity.secret
  ```

  ```bash Show public identity only
  # Extracts and prints the public portion (safe to share)
  zerotier-idtool getpublic identity.secret
  ```

  ```bash Sign a file
  # Signs file.bin using the identity's private key
  zerotier-idtool sign identity.secret file.bin
  ```

  ```bash Verify a signature
  # Verifies signature.bin against file.bin using identity.public
  zerotier-idtool verify identity.public file.bin signature.bin
  ```
</CodeGroup>

<Note>
  `zerotier-idtool generate` may take several seconds on first run because it must find a key pair that satisfies the hashcash condition. This is normal and expected.
</Note>

## Backing Up and Moving Identities

A node's ZeroTier address is defined entirely by `identity.secret`. To move a node to new hardware, or to ensure you can restore a node after a system failure:

<Steps>
  <Step title="Back up identity.secret">
    Copy `identity.secret` to secure offline storage. Losing this file means losing the node's address — a new identity and new address will be generated on next start.
  </Step>
  <Step title="Stop the ZeroTier service">
    Ensure the service is not running before copying identity files to avoid partial reads.
  </Step>
  <Step title="Copy to the new system">
    Place `identity.secret` (and optionally `identity.public`) in the ZeroTier working directory on the destination system.
  </Step>
  <Step title="Start the service">
    ZeroTier will read the existing identity files and resume using the same address. If the node was a member of any networks, it will re-authenticate with the same credentials.
  </Step>
</Steps>

<Warning>
  Do not run two ZeroTier instances with the same identity simultaneously. Duplicate identities on the network cause authentication errors and undefined behaviour. The node detects this via `ZT_EVENT_FATAL_ERROR_IDENTITY_COLLISION` and reports it as a fatal event (`include/ZeroTierOne.h:493`).
</Warning>

## Security Implications

| Property | Detail |
|----------|--------|
| **Key algorithm** | Curve25519 (ECDH key agreement) + Ed25519 (signatures) |
| **Address space** | 2^40 addresses (~1 trillion). Collision probability is negligible given the hashcash construction. |
| **Private key exposure** | If `identity.secret` is compromised, all past and future traffic to that address can be decrypted by an attacker who also captured the ciphertext. |
| **Key rotation** | There is no built-in key rotation. A new identity means a new address. Network administrators must re-authorize the new address on every network. |
| **Destruction** | When a `Peer` object is destroyed, its symmetric key material is immediately zeroed via `Utils::burn()` (`node/Peer.hpp:50`). The private key in `Identity` is similarly zeroed on destruction (`node/Identity.hpp:61`). |

The `Identity::agree()` method (`node/Identity.hpp:185`) performs ECDH key agreement between two identities to derive the 256-bit symmetric session key used for all subsequent packet encryption between those two nodes:

```cpp
inline bool agree(const Identity& id, void* const key) const
{
    if (_privateKey) {
        ECC::agree(*_privateKey, id._publicKey, key, ZT_SYMMETRIC_KEY_SIZE);
        return true;
    }
    return false;
}
```

This key is stored in `Peer::_key` and is used — along with AES keys derived via KBKDF — for all encrypted packets exchanged with that peer.
